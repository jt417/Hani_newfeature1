# 관계맵 가독성 개선 기획안

## 1. 문서 목적

현재 HANI의 관계맵은 시각적으로 매력은 있지만, 실제 사용 시 `한눈에 이해하기 어렵고`, `조작이 직관적이지 않으며`, `데이터가 늘수록 더 복잡해질 가능성`이 있다.

이 문서는 현재 구현을 기준으로 관계맵의 가독성과 탐색성을 개선하기 위한 방향을 정리한다.

핵심 목표는 아래 3가지다.

- 첫 진입 3초 안에 맵 구조를 이해하게 만들기
- 모바일에서 손쉽게 원하는 사람/그룹을 찾게 만들기
- 데이터가 늘어나도 화면이 덜 복잡해 보이게 만들기

---

## 2. 현재 구현 기준 진단

현재 관계맵은 아래 구조를 갖고 있다.

- 고정 크기 `1200 x 1200` 캔버스를 확대/축소해서 탐색한다.
- `나`는 하단 중앙 고정이고, 그룹은 상단/중앙에 원형 영역으로 배치된다.
- 인물과 그룹의 좌표는 데이터에 하드코딩되어 있다.
- 포커스 전환, 더블탭 상세 진입, 수동 줌, 수동 스크롤에 많이 의존한다.

### 2.1 주요 불편 요인

#### A. 구조를 이해하기 전까지 화면이 복잡해 보인다

- 그룹 원, 가지 선, 사람 노드, 이름 라벨, 뿌리, 연결선이 동시에 보인다.
- 처음 보는 사용자는 `무엇을 먼저 봐야 하는지` 알기 어렵다.
- 배경 장식과 구조선이 예쁘지만, 정보 계층이 강하지 않다.

#### B. 현재 배치는 모바일에서 시선 이동이 크다

- `나`가 맵 하단에 있고 그룹은 위쪽으로 흩어져 있어 손과 시선 이동이 길다.
- 확대 상태가 맞지 않으면 처음부터 맵 전체를 파악하기 어렵다.
- 그룹 간 여백은 큰데, 실제 핵심 정보는 상대적으로 작게 보인다.

#### C. 하드코딩 좌표 구조라 데이터가 늘면 더 답답해진다

- 현재 그룹 좌표는 `cx`, `cy`, 인물 좌표는 `x`, `y`로 직접 고정되어 있다.
- 인물 수가 늘어나면 라벨 겹침, 원 바깥 이탈, 선 겹침이 심해질 가능성이 높다.
- 그룹 추가 시 빈 슬롯에 넣는 방식은 빠르지만 레이아웃 품질이 일정하지 않다.

#### D. 포커스/상세 진입 방식이 직관적이지 않다

- 싱글탭은 포커스, 더블탭은 상세 진입인데 이 패턴은 모바일에서 발견성이 낮다.
- 사용자는 "한 번 눌렀는데 아무것도 안 열린다"고 느낄 수 있다.
- 하단 미니 시트가 뜨긴 하지만 맵과의 관계를 이해하기 전에는 행동이 분산된다.

#### E. 줌과 스크롤이 탐색의 기본이어서 피로하다

- 현재는 확대/축소와 스크롤을 직접 조절해야만 구조를 읽을 수 있다.
- 검색했을 때 자동으로 해당 노드가 가운데로 오지 않으면 탐색 효율이 떨어진다.
- "전체 보기", "선택 그룹만 보기", "선택 인물 중심 보기"가 명확하게 단계화되어 있지 않다.

#### F. 모든 노드가 비슷하게 보여 중요도가 약하다

- 이름, 원형 아바타, 배지 톤이 유사해서 무엇이 중요한 관계인지 직관적으로 드러나지 않는다.
- 정밀 해설 해금, NEW, pending 상태는 배지로 표시되지만 `관계 상태`, `최근 감정`, `회복/소모` 정보는 맵에서 약하다.

## 3. 문제를 한 줄로 정리하면

현재 관계맵은 `아름다운 구조도`에는 가깝지만, `빠르게 읽히는 지도`에는 덜 가깝다.

즉, 보기 좋은 그래프가 아니라 `찾기 쉬운 지도`로 재설계해야 한다.

## 4. 개선 방향 핵심 원칙

### 4.1 기본 원칙

- 기본 화면은 설명 없이도 읽혀야 한다.
- 디테일은 필요할 때만 단계적으로 보여줘야 한다.
- 모든 줌 레벨에서 보여주는 정보량이 달라야 한다.
- 모바일 한 손 사용을 기준으로 조작해야 한다.
- 관계맵은 "탐색"보다 "판단"이 빨라야 한다.

### 4.2 UX 원칙

- Overview first
- Focus second
- Detail on demand

즉, 기본은 전체 흐름, 다음은 인물/그룹 포커스, 마지막이 상세 페이지여야 한다.

## 5. 제안하는 정보 구조

관계맵은 한 개 화면이 아니라 아래 3개 레벨로 이해하는 편이 좋다.

### 5.1 Level 1. 전체 개요 맵

사용자가 처음 보는 화면

- 그룹이 몇 개 있는지
- 요즘 어느 그룹이 편안한지/예민한지
- 어떤 관계가 최근 많이 떠올랐는지

여기서는 `모든 사람의 상세 라벨`을 보여줄 필요가 없다.

### 5.2 Level 2. 그룹 포커스 맵

특정 그룹을 눌렀을 때 들어가는 상태

- 그 그룹 안에서 누가 핵심 인물인지
- 최근 감정이 어떤 방향으로 쏠리는지
- 그룹 전체 분위기가 어떤지

여기서는 해당 그룹 인물만 크게 보이고, 나머지 그룹은 축약한다.

### 5.3 Level 3. 인물 포커스

특정 인물을 눌렀을 때

- 나와의 연결선
- 관련 인물
- 최근 기록
- 관계 상태 요약

여기서만 하단 시트나 상세 액션을 적극적으로 보여준다.

## 6. 레이아웃 개선안

### 6.1 현재 트리형 배치의 한계

현재는 `나 -> 가지 -> 그룹 -> 인물` 구조다.

이 방식의 장점

- 상징성이 있다
- 브랜딩적으로 기억에 남는다

한계

- 하단 뿌리에서 상단 그룹으로 시야가 길게 찢어진다
- 그룹 간 밀도 차이가 커지면 보기 불편해진다
- 노드가 늘수록 빈 공간과 혼잡 공간이 함께 생긴다

### 6.2 추천 배치 방식: 하이브리드 클러스터 맵

권장 방식은 `나를 중심으로 그룹이 섬처럼 둘러싸는 클러스터형 배치`다.

예시 구조

```text
        [가족]

 [학교]   [나]   [친구]

        [직장]
        [모임]
```

이 방식의 장점

- 시선 중심이 화면 가운데로 모인다
- 그룹을 하나의 단위로 인식하기 쉽다
- 모바일에서도 첫 화면 이해가 빠르다

### 6.3 그룹 표현 방식 변경

현재의 큰 점선 원은 영역을 설명하기는 좋지만 실제로는 정보 밀도를 떨어뜨린다.

추천 변경

- 기본 개요 맵에서는 `큰 점선 원` 대신 `그룹 카드/캡슐`
- 그룹 포커스 모드에서만 확장 영역 표시
- 그룹별 색/톤 차는 아주 약하게만 사용

### 6.4 인물 배치 방식 변경

기존

- 인물 좌표를 수동으로 지정

개선

- 그룹 안에서 자동 배치
- 최근 상호작용이 많은 인물은 가운데, 적은 인물은 바깥쪽
- 너무 많을 경우 `+N명 더보기`로 축약

예시

- 기본: 그룹당 최대 4명만 노출
- 나머지: `+3`
- 그룹 포커스 진입 시 전체 확장

### 6.5 자동 정렬 규칙 제안

- 그룹 위치는 그룹 수에 따라 방사형 자동 배치
- 노드는 그룹 중심을 기준으로 원형 또는 반원형 배치
- 라벨은 충돌 감지 후 일부 생략
- 검색 결과는 자동 센터링

## 7. 시각 정보 계층 개선안

### 7.1 지금보다 줄여야 하는 요소

- 전체 연결선 수
- 기본 화면의 라벨 수
- 큰 장식성 원
- 동시에 보이는 상태 배지 수

### 7.2 기본 화면에서 강조해야 하는 요소

- 그룹 이름
- 그룹 상태
- 최근 많이 기록된 인물
- 내가 지금 봐야 할 관계

### 7.3 노드 시각 규칙 제안

#### 노드 크기

- 크기 = 최근 기록 빈도 또는 중요도

#### 노드 외곽선

- 외곽선 색 = 관계 상태
  - 편안함: 초록 계열
  - 예민함: 앰버 계열
  - 거리감: 회색/블루 계열
  - 회복: 민트 계열

#### 노드 내부

- 기본 상태: 이니셜
- 고정 배지 최소화
- NEW, 해금, pending은 아이콘 크기를 줄이고 우선순위 낮춤

#### 라벨

- 기본 개요 맵: 중요 인물만 이름 노출
- 그룹 포커스: 그룹 인물 전체 노출
- 인물 포커스: 선택 노드와 연결 인물만 선명하게 노출

### 7.4 연결선 표현 규칙 제안

기본 개요 맵에서는 모든 사람-사람 선을 보여줄 필요가 없다.

추천

- 기본 화면: `나 -> 그룹`, `그룹 -> 대표 인물`만 노출
- 인물 포커스: 선택 인물 관련 선만 노출
- 그룹 포커스: 해당 그룹 내부 선만 노출

이렇게 하면 선의 의미가 살아난다.

## 8. 탐색 인터랙션 개선안

### 8.1 더블탭 제거

현재의 더블탭 상세 진입은 발견성이 낮다.

추천

- 1탭: 하단 프리뷰 시트 열기
- 프리뷰 시트의 CTA 버튼: `상세 보기`, `AI`, `기록 보기`

이렇게 바꾸면 사용자는 행동 결과를 즉시 이해한다.

### 8.2 검색 동작 개선

검색은 필터만 할 것이 아니라 `바로 찾아주는 기능`이어야 한다.

추천

- 검색 결과가 1개면 자동 센터링
- 검색 결과가 여러 개면 해당 노드만 강조
- 검색 결과 없으면 빈 상태를 맵 안에서도 명확하게 표시

### 8.3 줌 동작 개선

기본 줌은 사용자가 맞추는 것이 아니라 시스템이 맞춰줘야 한다.

추천

- 첫 진입 시 전체 그룹이 보이도록 auto-fit
- 그룹 선택 시 해당 그룹만 보이도록 auto-zoom
- 인물 선택 시 해당 인물과 연결선이 보이도록 auto-center

### 8.4 컨트롤 위치 개선

현재 우측 하단 줌 컨트롤은 무난하지만, 맵 읽기와 관련된 핵심 액션은 더 가까워야 한다.

추천

- 상단: 전체 / 그룹 / 사람 보기 토글
- 하단: 선택 노드 액션
- 우측: 줌

### 8.5 미니맵 또는 위치 복귀 버튼

데이터가 늘어나면 현재 위치를 잃기 쉽다.

추천

- 간단한 미니맵
- 혹은 `전체 맞춤`, `내 위치`, `선택 그룹 맞춤` 3버튼

## 9. 관계 정보 오버레이 개선안

관계맵이 단순 구조도가 아니라면, 최근 데이터가 맵에서 읽혀야 한다.

### 9.1 최근 감정 흔적 오버레이

노드 주변에 최근 감정 흔적을 얇은 링으로 표시한다.

예시

- 초록 링: 안정/회복
- 핑크 링: 설렘/애정
- 앰버 링: 예민/긴장
- 회색 링: 거리감/무감

### 9.2 기록량 오버레이

인물 카드에 `기록 4건` 같은 숫자를 작은 배지로 두되, 기본 화면에서는 중요한 인물만 보여준다.

### 9.3 그룹 컨디션 오버레이

그룹 카드 배경 톤으로 그룹 상태를 약하게 보여준다.

- 안정
- 활발
- 예민
- 거리감

## 10. 모바일 기준 보기 편하게 만드는 구체적 방법

### 10.1 기본 화면에서 모든 이름을 보여주지 않는다

이름 라벨이 많아지면 가장 먼저 맵이 지저분해진다.

추천

- 기본 개요 맵: 그룹 대표 인물만 이름 노출
- 선택 그룹: 그룹 인물 전체 노출
- 인물 포커스: 관련 인물만 노출

### 10.2 배경 장식은 줄이고 정보 대비를 올린다

현재 도트 배경, 가지 라인, 그룹 원이 함께 있어서 정보층이 고르게 보인다.

추천

- 배경 도트 약화
- 가지 라인 두께 축소
- 그룹 원 대신 소프트 블롭 또는 카드

### 10.3 한 번에 보여주는 노드 수를 제한한다

초기 화면은 `전부 보여주는 것`보다 `읽히는 것`이 중요하다.

추천

- 기본 화면 노출: 그룹당 핵심 인물 3~4명
- 추가 인물은 `+N`
- 탭하면 확장

### 10.4 하단 시트 높이 고정

선택 시 뜨는 미니 프로필은 좋지만 맵을 가리는 높이가 크면 탐색이 끊긴다.

추천

- 1차 프리뷰는 낮은 높이
- 상세 정보는 별도 페이지

## 11. 빠르게 적용 가능한 개선안

아래는 현재 구조를 크게 바꾸지 않고도 먼저 할 수 있는 항목이다.

### P0. 즉시 개선

- 첫 진입 시 auto-fit 적용
- 검색 결과 자동 센터링
- 더블탭 제거, 단일 탭 프리뷰 전환
- 기본 화면 연결선 수 축소
- 그룹 점선 원 크기 축소
- 배경 도트/라인 대비 약화
- 미니 범례 접기 가능하게 변경

## 15. 최종 제안

관계맵을 보기 편하게 만드는 핵심은 `더 많은 정보를 넣는 것`이 아니라 `처음에는 덜 보여주고, 필요할 때만 더 보여주는 것`이다.

가장 중요한 변화는 아래 4가지다.

1. 고정 트리형 구조를 `클러스터형 개요 맵`으로 완화한다
2. 기본 화면의 선, 원, 라벨 수를 줄인다
3. 더블탭 대신 단일 탭 + 프리뷰 CTA로 바꾼다
4. 검색, 줌, 포커스를 시스템이 더 적극적으로 보조한다

결론적으로 관계맵은 `예쁜 그래프`보다 `빠르게 읽히는 관계 지도`가 되어야 한다.

## 20. 다중 그룹 멤버 UX/UI 최종안

이 섹션은 한 사람이 여러 그룹에 속할 때 관계맵에서 어떻게 보여주는 것이 가장 좋은지에 대한 UX/UI 기준 최종 제안이다.

예시 인물

- 조멀티 -> `g1(A프로젝트 TF)`, `g2(주말 러닝크루)`
- 윤소영 -> `g3(동네 베프들)`, `g4(대학 과 동기)`
- 송재원 -> `g1(A프로젝트 TF)`, `g3(동네 베프들)`

### 20.1 결론

UX/UI 관점에서 가장 좋은 방식은 `primary + ghost`가 아니라 `1인 1노드 + 브리지 존 배치`다.

즉, 다중 그룹 멤버는 화면에 여러 번 복제하지 않고, `하나의 노드로만 표시`하되 `소속된 그룹 사이의 중간 지점`에 배치하는 것이 가장 읽기 쉽다.

### 20.2 왜 이 방식이 가장 좋은가

#### A. 중복 노드가 없어 혼란이 적다

- 같은 사람이 두 번 보이면 사용자는 먼저 "왜 복제됐지?"를 해석해야 한다.
- 지도는 해석보다 인지가 빨라야 하므로, 기본 화면에서는 한 사람을 한 번만 보여주는 것이 유리하다.

#### B. 다중 소속의 의미가 위치로 바로 읽힌다

- 두 그룹 사이에 놓인 노드는 설명 없이도 "이 사람은 두 그룹을 잇는구나"라고 이해할 수 있다.
- 이것이 다중 그룹 멤버의 핵심 의미와 가장 잘 맞는다.

#### C. 개요 화면이 깔끔하게 유지된다

- ghost 노드, 점선 테두리, 브릿지 라인, 중복 라벨이 없으므로 첫 화면 복잡도가 크게 낮아진다.
- 모바일에서 특히 중요하다.

#### D. 브리지 역할이 더 자연스럽다

- 다중 그룹 멤버는 단순히 소속이 많은 사람이 아니라 `관계 구조를 잇는 사람`으로 보이는 편이 좋다.
- 위치 기반 표현이 이 의미를 가장 잘 전달한다.

### 20.3 모드별 추천 처리 방식

### 개요 모드

- 사람은 무조건 1번만 표시
- 단일 그룹 멤버는 해당 그룹 내부에 배치
- 다중 그룹 멤버는 관련 그룹 사이의 `브리지 존`에 배치
- 우하단 배지로 소속 그룹 수 표시
  - 예: `2`, `3`
- 필요하면 외곽 링을 2분할 또는 3분할해 소속 그룹을 약하게 암시

### 그룹 포커스 모드

- 선택한 그룹 안에서는 해당 인물을 일반 멤버처럼 자연스럽게 표시
- 다른 그룹 인스턴스를 ghost로 다시 보여주지 않음
- 대신 작은 배지나 텍스트로 `다른 그룹에도 속함`을 알려줌

예시

- `주말 러닝크루 외 1개 그룹`

### 노드 포커스 모드

- 선택 인물은 하나만 강조
- 소속된 그룹 전체를 동시에 하이라이트
- 하단 시트 또는 상세 영역에서 소속 그룹 목록을 명확히 노출

예시

- `소속 그룹: A프로젝트 TF · 주말 러닝크루`

### 20.4 시각적 표현 추천

#### 기본 노드

- 일반 노드 스타일 유지
- 다중 그룹 멤버만 우하단 숫자 배지 추가

#### 다중 그룹 멤버 배지

- `2`, `3`처럼 간단한 숫자
- 의미는 `소속 그룹 수`
- 과한 아이콘보다 숫자가 빠르게 읽힌다

#### 외곽 링

- 선택 사항
- 2개 그룹 소속이면 외곽 링을 2색 분할
- 3개 그룹 이상이면 너무 복잡해질 수 있어 기본은 단색 + 숫자 배지가 안전

#### 브리지 존

- 두 그룹의 정중앙이 아니라, 화면 균형을 고려한 `완만한 중간 지점`
- 그룹과 겹치지 않도록 약간 바깥쪽 아치 배치가 더 좋다

### 20.5 `primary + ghost` 방식이 최선이 아닌 이유

`primary + ghost`는 개념적으로는 정확하지만 UX/UI에서는 기본값으로 과하다.

#### 단점

- 같은 사람이 중복 노출된다
- 고스트의 의미를 학습해야 한다
- 브릿지 라인까지 생기면 선 밀도가 높아진다
- 다중 그룹 멤버가 많을수록 맵이 네트워크 다이어그램처럼 복잡해진다
- 모바일 첫 화면에서 직관성이 떨어진다

#### 적합한 경우

- 브리지 역할을 분석적으로 깊게 보고 싶을 때
- 기본 개요가 아니라 `고급 보기 모드`일 때

즉, ghost 방식은 기본 구조가 아니라 선택적 토글로만 쓰는 편이 낫다.

### 20.6 대안 비교

#### 대안 1. 1인 1노드 + 브리지 존 배치

장점

- 가장 읽기 쉽다
- 가장 깔끔하다
- 다중 그룹의 의미가 위치로 전달된다

단점

- 그룹 내부에 딱 맞춰 정렬되지는 않는다

#### 대안 2. 1인 1노드 + 대표 그룹 고정 + 멀티 그룹 배지

장점

- 가장 단순하다
- 화면 구조가 안정적이다

단점

- 다중 그룹 멤버의 "연결자" 성격이 약하게 표현된다

#### 대안 3. primary + ghost

장점

- 소속 관계를 가장 직접적으로 보여준다

단점

- 개요 화면이 복잡해진다
- 모바일 가독성이 떨어진다

### 20.7 최종 추천안

#### 기본값

- `1인 1노드`
- 다중 그룹 멤버는 `브리지 존` 배치
- 숫자 배지로 그룹 수 표시

#### 보조 표현

- 포커스 시 소속 그룹 목록 노출
- 필요 시 외곽 링 또는 색 분할로 보조 표시

#### 선택 기능

- 추후 `브리지 멤버만 보기` 또는 `연결 구조 보기` 같은 분석 모드 추가 가능
- 그때만 ghost나 브릿지 라인 같은 고급 표현을 제한적으로 사용할 수 있음

### 20.8 한 줄 정리

다중 그룹 멤버는 `복제해서 보여주는 대상`이 아니라 `여러 그룹을 잇는 하나의 사람`으로 보여줘야 한다.

관계맵의 기본 목표가 가독성이라면, 최선의 UX/UI는 `1인 1노드 + 브리지 존 배치`다.
