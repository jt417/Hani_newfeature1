# 관계 맵 / 다이어리 기능 상세 v1

## 1. 문서 목적

이 문서는 관계 맵, 다이어리, 그룹/커플 궁합 방, 정밀 궁합 해설 기능을 개발자가 실제 구현할 수 있도록 현재까지 확정된 정책과 동작 규칙을 정리한 기능 상세 문서다.

본 문서는 사용자 답변 기준으로 작성되었으며, 미확정 항목은 마지막 `추가 확인 필요`에 따로 분리했다.

## 2. 제품 전제

### 2.1 서비스 형태

- 기존 앱이 이미 존재한다.
- 관계 맵과 다이어리 기능은 기존 앱에 추가되는 기능이다.
- 로그인 기반 서비스다.

### 2.2 실행 환경

- 현재는 웹뷰로 띄운다.
- 앱 래퍼를 씌워 인터넷이 없는 상태에서도 기존 저장 정보를 볼 수 있어야 한다.
- 즉, 네트워크가 끊겨도 최소한 `조회`는 가능해야 한다.

### 2.3 프라이버시 기본 정책

- 관계 맵, 다이어리, 개인 결과는 기본적으로 본인 앱에서만 확인 가능하다.
- 다이어리는 완전 개인 영역이다.
- 법률 및 정책상 의무 보관 기간이 존재한다.

## 3. 엔티티 정의

### 3.1 관계 맵 기본 엔티티

- 나
- 사람
- 그룹

### 3.2 사람 엔티티 기본 규칙

- 한 사람은 여러 그룹에 속할 수 있다.
- 사람은 `나와 한 번 이상 직접 궁합을 본 사람`만 추가된다.
- 사람 삭제 시 관계 맵에서는 제거되지만, 다이어리 기록 자체는 남는다.
- 시스템이 관계 상태를 자동 계산한다.

### 3.3 그룹 엔티티 기본 규칙

- 그룹은 여러 사람을 묶는 단위다.
- 그룹 분석 결과를 저장하면 그룹과 사람 모두 관계 맵에 추가된다.
- 저장 후 개별 사람 삭제는 가능하다.

## 4. 관계 맵 기능 상세

### 4.1 사람 추가 조건

- 직접 궁합을 본 이력이 있어야 관계 맵에 사람으로 추가 가능
- 그룹 방 분석 결과 저장 시 참여자도 함께 관계 맵에 추가

### 4.2 사람 삭제 정책

- 관계 맵에서 인물별 개별 삭제 가능
- 삭제해도 해당 인물과 연결된 다이어리 기록 내용은 유지
- 기록 속 태그/텍스트는 남고, 관계 맵 연결만 제거되는 방향으로 해석

### 4.3 다중 그룹 소속

- 한 사람은 여러 그룹에 속할 수 있음
- 관계 맵 UI에서는 다중 그룹 멤버를 지원해야 함
- 상세한 표현 방식은 별도 문서에서 `1인 1노드 + 브리지 존` 권장안으로 정리됨

### 4.4 관계 상태 계산

- 사용자가 수동으로 상태를 붙이지 않음
- 시스템 자동 계산
- 향후 계산 기준은 다이어리 기록, 상호작용, 그룹 분석 결과를 조합 가능

### 4.5 민감 인물 관리

- 숨김 가능
- 잠금 가능
- 삭제 가능

### 4.6 실명 정책

- 별명은 허용하지 않음
- 인물 이름은 실명 기준으로 간주

### 4.7 동명이인 처리 정책

이름이 같더라도 생년월일시가 다르면 다른 사람으로 처리한다.

#### 데이터 기준

- 동명이인은 서로 다른 `personId`를 가진 별도 인물이다.
- 이름이 같다고 자동 병합하지 않는다.
- 관계 맵, 다이어리, 궁합 결과 연결은 이름이 아니라 `personId` 기준으로 처리한다.

#### 생성 기준

- 같은 이름의 인물이 이미 존재해도, 생년월일시가 다르면 새 인물로 추가 가능하다.
- 사람 추가 시 동일 이름이 발견되면 사용자에게 아래 분기를 보여준다.

예시

- `기존 인물 선택`
- `동명이인으로 새 인물 추가`

#### UI 표기 기준

기본적으로는 `이름 + 최소 구분자` 형태로 표시한다.

우선순위

1. 이름
2. 출생연도 또는 생년월일 일부
3. 관계/그룹 라벨
4. 필요 시 출생 시간대

예시

- `김민지 · 95년생`
- `김민지 · 97년생`
- `김민지 · 95년생 · 회사`
- `김민지 · 95.03.12 오전`

#### 화면별 적용

##### 관계 맵 리스트

- 동명이인 구분자가 항상 함께 보여야 함

##### 다이어리 태그 선택

- 이름만 보여주면 안 됨
- 반드시 구분자 포함

예시

- `김민지 · 회사 · 95년생`
- `김민지 · 친구 · 97년생`

##### 상세 페이지

- 상단에서 인물 식별이 명확해야 함
- 이름
- 출생 정보
- 소속 그룹 또는 관계 라벨

#### 금지 규칙

- 이름만 같다고 자동 병합 금지
- 이름만 보고 기존 인물에 덮어쓰기 금지
- 동명이인 구분자 없는 선택 UI 금지

## 5. 정밀 궁합 해설 정책

### 5.1 기본 구조

- 기본 궁합 해설과 정밀 궁합 해설은 분리된다.
- 현재 기준 기본/정밀 차이는 우선 지금 수준을 유지한다.
- 다만 유료 가치 강화를 위해 별도 개선안 문서가 존재한다.

### 5.2 결제 정책

- 정밀 궁합 해설은 1회 결제 후 영구 해금
- 인물 단위 구매
- 추후 묶음 상품 가능성 있음

### 5.3 해금 후 저장 정책

- 결제 후 결과는 영구 저장
- 단, 본인 또는 상대방의 사주 원국 정보가 변경되면 현재 분석은 정확하지 않다는 경고 문구를 띄워야 함

예상 경고 문구

- `xx님의 사주정보가 바뀌어 현재 분석은 정확하지 않습니다. 다시 분석 하시겠습니까?`
- `비용 청구: 원래의 반값`

### 5.4 재분석 비용 정책

- 사주 원국 정보 변경 후 재분석 시 원래 가격의 반값 청구

### 5.5 정밀 해설 포함 방향

- 현재 사용 답변 기준으로는 `전반적 분석`이 필요
- 구체 섹션 설계는 별도 문서에서 고도화 가능

## 6. 다이어리 기능 상세

### 6.1 기본 정책

- 다이어리는 개인 영역이다.
- 다이어리는 `하루 1기록` 원칙으로 운영한다.
- 사진 첨부는 당장 제외한다.
- 감정 선택지는 고정형이다.
- 다만 감정 카테고리는 현재보다 더 다각화가 필요하다.

### 6.2 기록 단위

- 한 날짜에는 기록 1개만 존재할 수 있다.
- 사용자는 하루 동안 여러 개의 기록을 새로 만드는 것이 아니라, 그날의 기록을 `수정/보완/확장`하는 방식으로 사용한다.
- 즉, 분석 기준은 `순간 감정`이 아니라 `하루의 대표 감정과 대표 사건`이다.

### 6.3 기록 구조

다이어리는 하나의 기록이 두 단계로 자라는 구조로 본다.

#### 1단계: 오늘 기록

- 하루를 짧게 정리하는 기본 입력
- 오늘의 대표 감정
- 에너지 상태
- 대표 사건 또는 한 줄 요약
- 관련 사람/그룹 연결

#### 2단계: 상세 확장

- 같은 날짜의 기록을 더 깊게 보완하는 방식
- 사건 배경
- 감정 흐름
- 강도
- 상세 메모
- 사주 기반 분석 강화

즉, `체크인`, `빠른 기록`, `깊이 기록`을 따로 분리하지 않고, `하루 기록 1개를 점점 깊게 확장`하는 구조로 정리한다.

### 6.4 체크인 / 빠른 기록 정책

- `오늘의 체크인`과 `빠른 기록`은 별도 기능으로 분리하지 않는다.
- 둘은 하나의 기능으로 통합한다.
- 하루 첫 진입에서는 같은 기능을 `오늘의 체크인`처럼 보여줄 수 있다.
- 하지만 저장 단위는 동일한 `하루 기록`이다.

예시

- 첫 진입 카피: `오늘의 체크인`
- 일반 편집 상태: `오늘 기록 수정`

### 6.5 기록 생성 규칙

#### 오늘 날짜

- 오늘 날짜 기록이 없으면 새로 작성 가능
- 오늘 날짜 기록이 이미 있으면 새 기록 생성이 아니라 수정/확장으로 진입

#### 과거 날짜

- 과거 날짜도 기록 가능
- 과거 날짜 역시 날짜당 1개만 허용
- 이미 해당 날짜 기록이 있으면 수정/확장만 가능

### 6.6 날짜 입력 정책

- 오늘 기록 가능
- 과거 날짜 기록 가능
- 회고형 기록 기능 포함

### 6.7 사람/그룹 연결

- 다이어리 기록은 사람/그룹과 연결 가능
- 복수 선택 가능

### 6.8 분석 관점 정책

- 분석은 `하루 대표 기록` 기준으로 누적한다.
- 하루에 감정이 여러 번 바뀌더라도, 최종적으로 남기는 하나의 기록을 그날의 분석 기준으로 사용한다.
- 이렇게 해야 장기 패턴 분석에서 노이즈를 줄일 수 있다.

### 6.9 공유 정책

- 다이어리 자체는 완전 개인 영역
- 다만 `나의 형충합파해 작용에 대한 종합 분석`은 공유 가능성 열어둠
- 현재 기준 MVP에서는 다이어리 원문 공유 없음

## 7. 그룹/커플 궁합 방 기능 상세

### 7.1 구조

- 그룹 궁합과 커플 궁합은 같은 방 구조를 사용
- 출력 양식만 다름

### 7.2 방 성격

- 일회성 분석 후 종료
- 영구 보존형 방이 아님

### 7.3 동시 방 운영

- 동시에 여러 방을 열 수 없음
- 새 QR/코드를 쓰려면 새 방을 열어야 함
- 다만 방을 열어놓은 상태에서 참여자 입장을 기다리는 동안 앱 내 다른 액션은 가능

### 7.4 최대 인원

- 초기 MVP 기준 최대 5명

### 7.5 참여 기준

- 로그인 사용자만 사용 가능
- 비로그인 사용자는 불가
- QR을 일반인이 스캔하면 앱 설치 페이지로 이동

### 7.6 분석 실행 규칙

- 방장이 분석 시작 권한을 독점
- 모든 슬롯이 다 차지 않아도 분석 가능
- 예: 5인 방을 열었지만 4명만 들어온 상태면 4인 궁합으로 분석 가능
- 분석 시작 전에는 임시 결과나 미리보기 결과를 제공하지 않음

### 7.7 결과 반영 기준

- 분석 시작 시점에 방 안에 있던 사람은 결과에 포함
- 분석 후 누가 나가더라도 그 실행 결과에는 포함된 것으로 간주

### 7.8 결과 저장

- 그룹 + 참여자 사람 모두 관계 맵에 기본 저장
- 이후 개별 삭제 가능

### 7.9 동일 멤버 재분석 제한

- 같은 멤버 조합의 사람들을 다시 모으면 기존 궁합이 이미 존재한다고 안내
- 즉, 중복 분석 판정 기준은 `같은 멤버 조합`이다
- 그룹명, 방 이름이 달라도 멤버 구성이 같으면 중복으로 본다
- 멤버가 1명이라도 달라지면 새로운 분석으로 본다

### 7.10 결과 공개 범위

#### 공용 결과

- 참여자 모두 열람 가능

#### 개인 결과

- 본인만 열람 가능

## 8. QR / 참여코드 / 초대 정책

### 8.1 QR 코드

- 일회성
- 유효 시간 30분
- QR 유효 시간 표시 필요

### 8.2 참여 코드

- 같은 방을 가리키는 대체 접속 수단
- QR은 가까이 있는 사람용
- 참여 코드는 멀리 있는 사람용
- 참여 코드도 30분간만 유효

### 8.3 만료 정책

- 새 QR/코드 재발급 개념보다, 새 방을 열면 새 코드가 생기는 구조
- 기존 방의 QR/코드는 30분 만료 기준

### 8.4 링크 정책

- QR 스캔 시 앱 사용자는 바로 진입 가능
- 일반인은 앱 설치 페이지로 이동
- 명시적 초대 링크 지원 여부는 현재 답변상 QR/코드 중심으로 해석

### 8.5 비로그인 사용자

- 비로그인 사용자는 방 참여 불가

## 9. 분석 실행 / 갱신 정책

### 9.1 시작 권한

- 방장만 시작 가능

### 9.2 시작 조건

- 참여자가 들어오는 대로 시작 가능
- 전원 입력 완료 대기 필요 없음

### 9.3 자동 재계산

- 결과는 자동 재계산된다

### 9.4 버전 이력

- 결과 버전 이력은 필요 없음

### 9.5 재분석 제한

- 같은 멤버 조합으로는 중복 분석 불가
- 같은 멤버 조합을 다시 방에 모으면 기존 궁합 존재 안내 필요
- 멤버 조합이 달라지면 새 분석 가능

- 본인만 열람 가능

### 10.4 프라이빗 라벨 / 민감 관계

- 잠금

모두 지원 필요

### 10.5 그룹 분석 후 관계맵 반영

- 같이 그룹 사주를 본 사람은 서로의 사주맵에 추가됨

## 12. 오프라인/동기화 정책 초안

- 저장된 다이어리 조회 가능

- 결제/해금
- 자동 재계산

- 앱 래퍼 환경에서 로컬 캐시 필요
- 서버 데이터와 로컬 데이터의 동기화 충돌 정책은 별도 정의 필요

## 13. MVP 범위 해석

사용자 답변상 원칙은 아래다.

- 구현이 쉬운 것은 MVP
- 구현이 어려운 것은 후순위

현재 답변을 기반으로 한 현실적 MVP 해석

### MVP 포함

- 관계 맵 기본 구조
- 사람/그룹 저장
- 다이어리 기록
- 빠른 기록 / 깊이 기록 기본 구조
- 그룹/커플 방 생성
- QR/참여코드 30분 유효
- 공용 결과 / 개인 결과 구분
- 정밀 궁합 1회 결제 후 영구 해금
- 정보 변경 시 재분석 경고

### 후순위 가능

- 다이어리 공유형 종합 분석
- 묶음 상품
- 고급 감정 체계
- 고급 오프라인 충돌 처리

---

## 14. 현재 기준 추가 확인 필요

아래 항목은 답변이 비어 있거나 구현 해석이 갈릴 수 있어 2차 확정이 필요하다.

### 14.1 다이어리 공유 범위

답변상

- 다이어리는 완전 개인영역
- 그러나 종합 분석은 공유 가능성 있음

확인 필요

- 현재는 후순위로 미뤘으나, 이후 어떤 수준까지 공유를 허용할지 별도 정책 필요

### 14.2 법률상 보관 정책 세부

답변상

- 의무 보관 기간 존재

확인 필요

- 어떤 데이터가 몇 년 보관인지
- 사용자에게 즉시 삭제로 보이는지, 실제는 분리 보관인지

## 15. 재분석 비용 정책 확정

### 15.1 정밀 궁합 재분석

- 본인 정보 변경 시 반값 재분석 적용
- 상대 정보 변경 시 반값 재분석 적용
- 즉, 둘 다 포함

### 15.2 그룹 궁합 재분석

- 다른 멤버 조합은 새 분석 가능
